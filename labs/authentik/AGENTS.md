# Authentik Lab - AI Agent Guidelines

This document provides guidance for AI agents working with the Authentik OIDC integration lab.

## Lab Overview

This lab demonstrates HashiCorp Vault integration with Authentik (open-source identity provider) using OIDC authentication. It provides a complete automated setup for:

- Authentik 2026.2.2 with PostgreSQL backend
- Automated admin user creation and API token generation
- Terraform-managed Vault OIDC configuration
- User and group management via Terraform
- Policy-based access control

## Architecture

### Components
- **Authentik Server**: OIDC provider (port 9000)
- **PostgreSQL**: Database backend (internal)
- **Authentik Worker**: Background task processor
- **Vault**: OIDC client (configured via Terraform)

### Network Connectivity

**Critical**: Terraform and Vault require different URLs to reach Authentik:

1. **Terraform Provider** (runs on host):
   - Uses `authentik_url = http://localhost:9000`
   - Connects via port forwarding to host

2. **Vault OIDC** (runs in container):
   - Uses `vault_authentik_url = http://authentik-server:9000`
   - Connects via Docker service name on `docker-vault-stack` network

**Implementation**:
- `variables.tf` defines `vault_authentik_url` for Vault OIDC configuration
- `vault.tf` uses `var.vault_authentik_url` for OIDC discovery URLs
- `.env` sets `AUTHENTIK_URL=http://localhost:9000` for Terraform provider
- Vault container resolves `authentik-server` via Docker DNS

**Why This Matters**:
- Using `localhost` in Vault OIDC config fails (Vault can't reach host)
- Using `authentik-server` in Terraform provider fails (host can't resolve Docker service names)
- Both services are on the same Docker network and can communicate using service names

### Key Files
- `docker-compose.yml`: Authentik stack definition
- `setup-admin.sh`: Automated admin setup and token generation
- `Taskfile.yml`: Task automation (uses parent Taskfile's dotenv)
- `.env`: Environment configuration (REQUIRED)
- `*.tf`: Terraform configuration for Vault OIDC setup

## Environment Setup

### Required Environment Variables (.env)

```bash
# Authentik Configuration
AUTHENTIK_SECRET_KEY=<generate with: openssl rand -base64 32>
AUTHENTIK_BOOTSTRAP_TOKEN=<optional bootstrap token>

# PostgreSQL Configuration
PG_PASS=<generate random password>
PG_USER=authentik
PG_DB=authentik

# Version Pinning
AUTHENTIK_IMAGE=ghcr.io/goauthentik/server
AUTHENTIK_TAG=2026.2.2

# Admin User Configuration (REQUIRED for setup-admin.sh)
AUTHENTIK_ADMIN_USER=akadmin
AUTHENTIK_ADMIN_PASSWORD=<MUST BE SET - no default>

# Authentik Provider Configuration (used by Terraform)
AUTHENTIK_URL=http://localhost:9000
AUTHENTIK_TOKEN=<auto-generated by setup-admin.sh>

# Docker Compose
COMPOSE_PROJECT_NAME=docker-vault-stack
```

### Important Notes

1. **AUTHENTIK_ADMIN_PASSWORD is REQUIRED**: The `setup-admin.sh` script will fail if this is not set in `.env`
2. **No Redis Required**: Authentik 2024.2+ removed Redis dependency
3. **Dotenv Declaration**: This Taskfile cannot have `dotenv:` declaration as it's included by parent Taskfile
4. **Environment Sourcing**: Tasks that need latest .env values use `source .env &&` prefix

## Complete Workflow

### Option 1: Automated End-to-End Setup

```bash
# From project root - start all services
task up

# Run complete Authentik setup
task authentik:all
```

This runs:
1. `./setup-admin.sh` - Create admin user, set password, generate API token
2. `terraform init` - Initialize Terraform providers
3. `terraform apply -auto-approve` - Create Vault OIDC configuration

**Important**: If OIDC auth backends already exist in Vault, delete them first:
```bash
vault auth disable oidc
vault auth disable -namespace=admin oidc
task authentik:apply
```

### Option 2: Step-by-Step Setup

```bash
# From project root - start all services
task up

cd labs/authentik

# 1. Create admin user and generate API token
./setup-admin.sh
# This will:
# - Wait for Authentik to be ready (HTTP 200)
# - Create/update admin user with password from .env
# - Generate API token
# - Update .env with AUTHENTIK_TOKEN

# 2. Initialize Terraform
task authentik:init

# 3. Review Terraform plan
task authentik:plan

# 4. Apply Terraform configuration
task authentik:apply

# 5. Test OIDC authentication
task authentik:test-auth
```

### Option 3: Clean Slate Deployment

```bash
# Redeploy Authentik with fresh data
task authentik:redeploy

# Run complete setup
task authentik:all
```

## Setup Script Details

### setup-admin.sh Behavior

The `setup-admin.sh` script:

1. **Loads .env file**: Uses `source .env` to get latest environment variables
2. **Validates password**: Fails if `AUTHENTIK_ADMIN_PASSWORD` is not set
3. **Waits for HTTP 200**: Polls Authentik health endpoint (60 attempts, 2s interval)
4. **Creates/updates user**: Uses Django ORM to manage admin user
5. **Sets password**: Applies password from `AUTHENTIK_ADMIN_PASSWORD`
6. **Generates token**: Creates API token with admin permissions
7. **Updates .env**: Writes `AUTHENTIK_TOKEN` back to .env file

### Why AUTHENTIK_BOOTSTRAP_PASSWORD Doesn't Work

- `AUTHENTIK_BOOTSTRAP_PASSWORD` only works during **first-time database initialization**
- Has **no effect** on existing databases
- Cannot reset passwords for existing users
- Use `setup-admin.sh` instead for existing installations

## Troubleshooting

### Issue: setup-admin.sh fails with "AUTHENTIK_ADMIN_PASSWORD must be set"

**Solution**: Ensure `.env` file contains `AUTHENTIK_ADMIN_PASSWORD=<your-password>`

### Issue: Terraform asks for authentik_token variable

**Solution**:
1. Run `./setup-admin.sh` first to generate token
2. Verify `.env` contains `AUTHENTIK_TOKEN=<token>`
3. Ensure parent Taskfile has `dotenv: [.env]` declaration

### Issue: Authentik not ready after 60 attempts

**Solution**:
1. Check container logs: `task authentik:logs-server`
2. Verify PostgreSQL is healthy: `task authentik:logs-postgres`
3. Check Docker resources (memory, CPU)
4. Restart stack: `task authentik:restart`

### Issue: Docker Compose warnings about AUTHENTIK_SECRET_KEY

**Solution**: Ensure `.env` contains `AUTHENTIK_SECRET_KEY=<value>` (not `SECRET_KEY`)

### Issue: Task fails with "Included Taskfiles can't have dotenv declarations"

**Solution**: Remove `dotenv:` from `labs/authentik/Taskfile.yml` - it should be in parent Taskfile only

## Development Workflow

### Making Changes

1. **Modify Terraform**: Edit `*.tf` files
2. **Plan changes**: `task authentik:plan`
3. **Apply changes**: `task authentik:apply`
4. **Test**: `task authentik:test-auth`

### Cleaning Up Vault OIDC Configuration

If you need to completely remove the Vault OIDC configuration and Terraform state:

```bash
# Remove Terraform state and disable Vault OIDC auth
task authentik:purge
```

This task will:
1. Remove Terraform state for all OIDC-related resources (identity groups, auth backends, policies)
2. Disable the Vault OIDC auth backend
3. Delete the Vault identity group `authentik-vault-admin-external`

**Use cases**:
- Starting fresh with a new OIDC configuration
- Resolving state conflicts or drift issues
- Cleaning up before redeploying Authentik

### Adding New Users

1. Edit `terraform.tfvars`:
```hcl
authentik_users = {
  "newuser" = {
    display_name = "New User"
    email        = "newuser@example.com"
    password     = "SecurePassword123!"
    groups       = ["vault-user"]
  }
}
```

2. Apply changes: `task authentik:apply`

### Updating Policies

1. Edit `policies.tf`
2. Plan: `task authentik:plan`
3. Apply: `task authentik:apply`
4. Verify: `task authentik:check-policies`

## Security Considerations

1. **Never commit .env**: Contains secrets and tokens
2. **Use strong passwords**: For `AUTHENTIK_ADMIN_PASSWORD`
3. **Rotate tokens**: Regenerate API tokens periodically
4. **Review policies**: Ensure least-privilege access
5. **Monitor logs**: Check for unauthorized access attempts
